OBJECTIVES
Subagent fleet build-out
Purpose: Establish a reliable pattern of stack-specific subagents that deliver end-to-end backend changes with minimal handholding.
End State: Subagent definitions exist for the Nest monorepo and the Python/React SPA. Each has a documented brief format, mandatory verification gates, and an evidence bar Team trusts. Brief in → PR out.
Appetite: Initial bring-up over weeks; per-stack capability expands as recurring gaps surface.
No-gos: Not building one mega-subagent. Not letting subagents merge without verification gates. Not duplicating cross-cutting logic — extract to a shared rule first.
Backend end-to-end ownership
Purpose: Own the design and verification of backend flows that span services, schemas, and queues — the work no single repo's tests catch alone.
End State: Documented flow specs for the most-touched cross-service journeys, each backed by an integration or contract test that fails when the flow breaks. Outage-driven gaps generate new specs, not postmortem-only entries.
Appetite: Ongoing, weighted toward high-traffic flows. Plumbing changes get a lighter review; new feature flows get full treatment.
No-gos: Not blocking PRs on flow docs that don't exist yet — backfill, don't gate. Not boil-the-ocean documentation. Not owning frontend flows past the integration boundary.
Frontend sanity gates
Purpose: Catch obvious React-SPA regressions (broken build, type errors, smoke-path failures) without taking ownership of frontend design.
End State: Type check, build, and a single smoke E2E run as required gates on the SPA's PRs. Team comments on visible UX regressions but defers design calls to whoever owns frontend.
Appetite: Low-touch maintenance — invest only when something breaks, then encode the lesson.
No-gos: Not designing components. Not redrawing layouts. Not blocking a PR on subjective UI taste.
